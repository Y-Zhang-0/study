## W21（D141-147）：集群与高可用

### D141 — CAP 定理与一致性模型

- CAP 定理三要素（Consistency / Availability / Partition Tolerance），为什么只能三选二
- BASE 与 ACID 对比：适用场景、取舍思路
- 一致性模型分层：强一致 → 线性一致 → 因果一致 → 最终一致，每种模型的典型系统举例
- 结合已有 RAG 系统分析：ChromaDB 属于哪种一致性？Redis 缓存层属于哪种？

**检查点**：能画出 CAP 三角并标注 Redis Cluster / ChromaDB / PostgreSQL 各自的位置，说清各一致性模型的区别

### D142 — 无状态服务改造

- 有状态 vs 无状态服务的本质区别，为什么无状态是水平扩展的前提
- 三类状态外移策略：会话状态存 Redis、文件存对象存储（MinIO/S3）、配置从环境变量注入
- 12-Factor App 中与无状态相关的原则（进程、配置、日志、端口绑定）
- 改造检查清单：审查现有服务中哪些地方存在隐式状态（内存缓存、本地文件路径、全局变量）

**检查点**：列出当前 RAG 服务的状态点清单，每个点标注外移方案，确认改造后任意实例可被替换

### D143 — 分布式锁

- Redis 分布式锁核心流程：SETNX 获取 → Lua 脚本原子释放 → token 防误删
- TTL 超时陷阱：业务耗时超过锁 TTL 导致并发问题，看门狗续期方案
- Redlock 算法概述：多节点获取、时钟漂移容忍、争议与适用场景
- 与数据库乐观锁/悲观锁的对比：各自适用场景和性能特征

**检查点**：能在白板上画出分布式锁的获取-释放-超时流程，说清 token 防误删的原理

### D144 — Redis 集群搭建

- Redis Cluster 架构：16384 哈希槽分配、3 主 3 从最小集群、Gossip 协议
- 故障转移流程：从节点选举、槽迁移、客户端重定向（MOVED / ASK）
- Cluster vs Sentinel 对比：Sentinel 适合主从高可用，Cluster 适合数据分片+高可用
- 搭建步骤梳理：配置文件关键参数、节点发现、槽分配、验证集群状态

**检查点**：本地启动 3 主 3 从 Redis Cluster，`redis-cli --cluster info` 显示所有节点正常，能模拟一个主节点宕机后自动故障转移

### D145 — 分布式限流

- 限流算法对比：固定窗口 → 滑动窗口 → 漏桶 → 令牌桶，各自的优缺点
- Redis 分布式限流实现：滑动窗口用 ZSET、令牌桶用 Lua 脚本保证原子性
- 多实例共享限流：所有服务实例共用 Redis 计数器，避免单实例限流不一致
- 限流粒度设计：全局限流 / 用户级限流 / API 级限流，不同粒度的 key 设计

**检查点**：用 Redis 实现一个滑动窗口限流，多终端并发请求验证限流生效且计数准确

### D146 Sat 8h — 实战：Redis 集群 + 分布式锁 + 语义缓存升级

- 将 RAG 系统的 Redis 缓存层升级为 Redis Cluster 模式
- 在文档摄入流程中加入分布式锁，防止同一文档被并发重复摄入
- 语义缓存升级：相似查询命中缓存时走 Redis Cluster，验证跨节点读写一致性
- 压测验证：模拟多实例并发摄入同一批文档，确认无重复、无丢失

**检查点**：Redis Cluster 运行稳定，分布式锁防重生效，语义缓存命中率符合预期

### D147 Sun 8h — 实战：无状态服务改造 + 分布式限流验证

- 按 D142 改造清单完成无状态改造，启动两个服务实例验证请求可任意路由
- 接入分布式限流中间件，配置全局 + 用户级双层限流策略
- 负载测试：两个实例同时运行，验证限流计数在 Redis 层全局一致
- 故障演练：关闭一个实例，确认另一个实例无缝接管且限流状态不丢失

**检查点**：两实例部署可互换，限流全局一致，故障转移无感知

## W22（D148-154）：K8s + 系统设计 + Buffer 6

### D148 — K8s 核心概念

- K8s 架构总览：Master（API Server / Scheduler / Controller Manager / etcd）+ Worker（kubelet / kube-proxy）
- 核心资源对象：Pod（最小调度单元）→ Deployment（声明式副本管理）→ Service（服务发现与负载均衡）→ Ingress（外部流量入口）
- 配置与密钥：ConfigMap（非敏感配置）vs Secret（敏感信息，Base64 编码）
- 生命周期管理：滚动更新策略、健康检查（liveness / readiness / startup probe）

**检查点**：能画出 K8s 架构图，说清从外部请求到 Pod 的完整链路（Ingress → Service → Pod）

### D149 — K8s AI 工作负载

- 资源管理：requests vs limits 的含义，AI 推理服务的 CPU/Memory 合理配置
- HPA 水平自动扩缩：基于 CPU/Memory 指标自动调整 Pod 副本数，扩缩策略与冷却时间
- GPU 调度概念：nvidia-device-plugin、资源声明 `nvidia.com/gpu`、节点亲和性
- AI 推理服务特点：冷启动慢（模型加载）、内存占用大、请求延迟敏感，对应的 K8s 策略

**检查点**：能写出一个 AI 推理服务的 Deployment YAML（含资源限制和健康检查），说清 HPA 配置参数的含义

### D150 — Helm Charts

- Helm 核心概念：Chart（包结构）、Release（部署实例）、Repository（Chart 仓库）
- Chart 结构：`Chart.yaml` / `values.yaml` / `templates/` 目录组织
- 模板语法：Go template 基础、`{{ .Values.xxx }}`、条件判断、循环、helper 函数
- Release 管理：install / upgrade / rollback / uninstall，版本历史追踪

**检查点**：为 RAG 系统创建 Helm Chart 骨架，`helm template` 渲染输出正确的 K8s 资源清单

### D151 — 系统设计框架

- 七步法：需求澄清 → 容量估算 → API 设计 → 数据模型 → 高层架构 → 深入设计 → 权衡讨论
- 需求澄清技巧：功能需求 vs 非功能需求，关键数字（DAU、QPS、存储量、延迟要求）
- 容量估算：QPS 推导、存储增长、带宽计算，常用数字速查（磁盘/内存/网络量级）
- 权衡讨论：每个设计决策都有 trade-off，面试时主动提出替代方案展示深度

**检查点**：用七步法分析一个简单系统（如短链接服务），完成从需求到架构的全流程，每步产出清晰

### D152 — 系统设计实战：10 万用户 RAG 系统

- 需求定义：10 万日活用户，峰值 1000 QPS 查询，100 万文档存储，P99 延迟 < 3 秒
- 容量规划：向量存储空间、嵌入计算资源、LLM 推理吞吐、缓存命中率目标
- 架构设计：API Gateway → 负载均衡 → 无状态服务层 → 向量数据库集群 → LLM 推理池 → 缓存层
- 深入讨论：语义缓存策略、向量库分片方案、LLM 推理队列设计、降级与熔断策略
- 画架构图：标注每个组件的选型理由、数据流向、扩展策略

**检查点**：完成完整的架构图 + 容量规划表，能在 15 分钟内讲清楚全链路设计

### D153 Sat 8h — Buffer 6：K8s 本地部署验收

- 安装 minikube 或 k3s 本地集群
- 将 RAG 系统容器化：编写 Dockerfile、构建镜像、推送到本地 registry
- 使用 D150 创建的 Helm Chart 部署到本地集群
- 验证：Service 可访问、Pod 自动重启恢复、HPA 配置生效（模拟负载触发扩缩）
- 排查常见问题：镜像拉取失败、资源不足、端口冲突、探针配置不当

**检查点**：`kubectl get pods` 显示所有 Pod Running，通过 Ingress/NodePort 成功访问 RAG 服务

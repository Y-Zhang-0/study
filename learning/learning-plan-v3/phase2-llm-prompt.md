## Week 3 (D15-D21)：多厂商 LLM API

### D15 周一：Claude API 基础（Messages/system/streaming）（2h）

**学习内容**：
- Anthropic Messages API 的请求结构：model、max_tokens、messages 数组与 role（user/assistant）的交替约束
- system 提示的定位：作为独立顶层参数而非 message，用于设定角色、规则与输出约束
- 多轮对话的拼装方式：把历史 user/assistant 消息原样回传，模型本身无状态需客户端维护上下文
- 流式输出（streaming）原理：SSE 分块返回 content_block_delta 事件，逐字渲染降低首字延迟
- 关键参数语义：temperature（随机性）、top_p（核采样）、stop_sequences（停止符）的取值影响
- API Key 管理与计费基础：环境变量注入密钥、请求/响应 token 计费维度、避免密钥硬编码

**检查点**：
- [ ] 能说清 messages 数组的 role 交替规则与 system 参数的独立地位
- [ ] 理解流式与非流式两种返回模式的差异及各自适用场景

### D16 周二：Claude 结构化输出（tool_use/JSON）+ Function Calling 原理（2h）

**学习内容**：
- 结构化输出的两条路径：Prompt 约束输出 JSON（脆弱）vs tool_use 工具调用（强约束、推荐）
- tools 参数定义：name、description、input_schema（JSON Schema）三要素如何引导模型生成参数
- tool_use / tool_result 消息流：模型返回 tool_use 块 -> 客户端执行 -> 以 tool_result 回传结果继续对话
- Function Calling 的本质：模型不执行函数，只输出"调用意图与参数"，执行与回传由应用负责
- 强制工具选择：tool_choice 控制（auto / any / 指定工具）以保证必定产出结构化结果
- JSON Schema 描述质量对参数准确率的决定性影响：字段说明、枚举约束、必填项设计

**检查点**：
- [ ] 能解释 Function Calling 中"模型出意图、应用做执行"的职责边界
- [ ] 理解 tool_use 与 tool_result 在 messages 中的成对出现关系

### D17 周三：Claude Prompt Caching + 成本/Token 优化（2h）

**学习内容**：
- Prompt Caching 机制：对稳定的 system / 长上下文前缀打缓存断点，命中后大幅降低重复输入成本
- 缓存适用场景：长系统提示、固定知识片段、Few-shot 示例集等高复用前缀
- 缓存命中条件：前缀必须逐字节一致、缓存有生命周期（TTL），动态内容需放在前缀之后
- 成本结构拆解：输入 token、输出 token、缓存写入与缓存命中读取的不同计价
- Token 优化手段：压缩冗余上下文、复用缓存前缀、控制 max_tokens 上限、按需裁剪历史
- 成本分析方法：估算单次请求成本、按日活与轮次外推月度费用、识别高成本调用路径

**检查点**：
- [ ] 能说清 Prompt Caching 的命中条件与"稳定前缀在前、动态内容在后"的布局原则
- [ ] 理解缓存写入与命中读取的计价差异及收支平衡点
- [ ] 产出一份缓存策略 + 成本分析：标注哪些内容入缓存、预估优化前后成本

## Week 4 (D22-D28)：Prompt 工程精通

### D22 周一：Prompt 基础（Role/Few-shot/CoT/格式/分隔符）（2h）

**学习内容**：
- 角色设定（Role）：用 system 设定身份、能力边界与语气，约束模型行为基调
- 少样本示例（Few-shot）：通过输入输出范例引导格式与风格，零样本与少样本的取舍
- 思维链（Chain-of-Thought）：引导模型分步推理以提升复杂任务正确率，适用于多步逻辑
- 输出格式控制：明确要求 JSON / 列表 / 表格等结构，配合 Schema 约束减少解析失败
- 分隔符与结构化提示：用标签或分隔符隔离指令、上下文、示例，降低指令污染与歧义
- 指令清晰原则：具体、可执行、无歧义；正面指令优于否定指令；约束前置

**检查点**：
- [ ] 能用 Role + Few-shot + 格式约束写出稳定产出结构化结果的提示
- [ ] 理解 CoT 适用场景及其对 token 成本与延迟的影响
- [ ] 沉淀一份可复用的 Prompt 模板库（含角色、少样本、格式约束模板）

---

### D23 周二：Prompt 进阶（Self-consistency/ToT/Prompt 链/元提示）（2h）

**学习内容**：
- 自洽性（Self-consistency）：对同一问题多次采样取多数答案，提升复杂推理稳定性
- 思维树（Tree of Thoughts）：并行探索多条推理路径并评估剪枝，适合需要搜索的难题
- Prompt 链（Prompt Chaining）：把复杂任务拆成多步串联，每步聚焦单一子目标降低难度
- 元提示（Meta-prompting）：让模型生成/改写提示词本身，用模型优化模型的指令
- 进阶技巧的代价：多次采样与多路径探索显著增加 token 与延迟，需权衡收益
- 选型决策：简单任务用基础提示，多步推理用 CoT/链，高难度搜索类才上 Self-consistency/ToT

**检查点**：
- [ ] 能说清 Self-consistency 与 ToT 的差异及各自适用的任务类型
- [ ] 理解 Prompt 链相比单次大提示在可控性与可调试性上的优势
- [ ] 产出一组进阶 Prompt 示例（含自洽性投票或 Prompt 链分解）

### D24 周三：上下文窗口管理（长文本/截断/压缩/滑动窗口）（2h）

**学习内容**：
- 上下文窗口本质：模型一次能处理的 token 上限，超限即报错或丢失信息
- 截断策略：按优先级保留（system + 最近消息优先），裁剪旧对话或低相关内容
- 摘要压缩：对历史对话做滚动摘要，用摘要替代原文以压缩 token 占用
- 滑动窗口：只保留最近 N 轮对话原文 + 早期摘要，平衡上下文完整性与成本
- 长文档处理：分块 + 按需召回相关片段（为 Phase 4 RAG 埋下检索式上下文的伏笔）
- 上下文预算分配：在窗口内为 system / 历史 / 当前输入 / 输出预留各自配额

**检查点**：
- [ ] 能为多轮对话设计"system + 滑动窗口 + 滚动摘要"的上下文管理方案
- [ ] 理解截断、压缩、召回三类手段的适用边界与代价
- [ ] 产出一份上下文管理方案：明确预算分配与超限时的降级处理

### D25 周四：幻觉治理（基于资料回答/引用/置信度/自检）（2h）

**学习内容**：
- 幻觉成因：模型基于概率续写、缺乏事实约束、上下文不足或被诱导时易编造内容
- 基于资料回答（grounding）：明确要求"只依据提供的资料作答，资料无则说不知道"
- 引用与可溯源：要求标注来源片段/编号，让答案可被核验，抑制无依据输出
- 置信度表达：让模型对答案给出确定性等级或区分"已知/推测/未知"
- 自检与反思：让模型对初稿做事实核对与自我批评（self-critique）后再定稿
- 局限与组合：Prompt 手段只能降低而非消除幻觉，需与检索（RAG）、规则校验组合使用

**检查点**：
- [ ] 能写出"基于资料回答 + 强制引用 + 无依据则拒答"的幻觉控制 Prompt 模板
- [ ] 理解置信度表达与自检反思在抑制幻觉中的作用与局限
- [ ] 产出可复用的幻觉控制 Prompt 模板（含 grounding、引用、自检三段）

- 评估闭环：评估 -> 定位瓶颈 -> 改 Prompt -> 再评估的迭代流程

**检查点**：
- [ ] 能为一类任务定义至少三项可量化的 Prompt 评估指标

## Week 5 (D29-D35)：Function Calling + Token 经济学 + Buffer

### D29 周一：Function Calling 深入（并行调用/工具选择/参数校验）（2h）

**学习内容**：
- 并行工具调用：模型一次返回多个 tool_use，应用并发执行后批量回传，降低多步延迟
- 工具选择准确率：工具数量、命名、描述质量直接影响模型选对工具的概率
- 工具集设计：避免功能重叠、控制工具数量、用清晰描述与 Schema 引导正确选择
- 参数校验：模型生成的参数不可信，应用侧必须做类型/范围/必填校验并对错误反馈重试
- 多轮工具循环：tool_use -> 执行 -> tool_result -> 模型再决策的循环及最大轮次保护
- 工具执行健壮性：结构化错误返回、超时、幂等，把异常以结果形式回传供模型纠偏

**检查点**：
- [ ] 理解并行工具调用的执行与回传流程及其延迟收益
- [ ] 能通过工具描述与 Schema 优化提升模型工具选择准确率
- [ ] 完成一个含并行调用 + 参数校验的 Function Calling 示例

### D30 周二：多模态输入（Vision/图像理解/OCR）（2h）

**学习内容**：
- 多模态消息结构：在 message 中混合文本与图像块（base64 或 URL），模型联合理解
- 图像理解能力边界：描述、问答、图表解读、布局识别的可用度与典型失败模式
- OCR 与文档解析：用 Vision 直接读图中文字，与传统 OCR 方案的取舍（精度/成本/版式）
- 多模态成本：图像按分辨率折算 token，大图成本高，需压缩/裁剪控制开销
- 提示技巧：明确指出关注的图像区域与任务目标，结构化要求输出降低歧义
- 应用场景：截图问答、票据/表单信息抽取、图文混排文档理解（衔接 Phase 4 多格式摄入）

**检查点**：
- [ ] 能构造文本 + 图像混合的多模态请求并获得正确理解结果

### D31 周三：Token 经济学（计数/成本/限流/批处理）（2h）

**学习内容**：

- 限流应对：理解 RPM/TPM 速率限制，用令牌桶/排队/退避平滑请求避免触发限流

### D32 周四：LLM 安全（提示注入/内容过滤/越狱防御）（2h）

**学习内容**：
- 提示注入（Prompt Injection）：用户输入或外部内容篡改/覆盖系统指令的攻击原理
- 间接注入：被检索/读取的外部文档中藏匿恶意指令，在 RAG/Agent 场景尤需警惕
- 防御手段：指令与数据隔离、不信任外部内容、用分隔符标注、最小权限工具授权
- 内容过滤：输入侧拦截违规请求、输出侧过滤敏感/有害内容，前后置双重护栏
- 越狱防御：识别角色扮演绕过、编码混淆等越狱手法，强化系统约束与拒答策略
- 纵深防御理念：单一手段不足，需"输入校验 + 指令隔离 + 输出过滤 + 权限控制"叠加

**检查点**：
- [ ] 能解释直接注入与间接注入的差异及在 RAG/Agent 中的风险点
- [ ] 理解"指令与数据隔离 + 不信任外部内容"的核心防御原则
- [ ] 产出一份 LLM 安全检查清单（注入/内容过滤/越狱/权限分层）

### D33 周五：API 最佳实践（重试/退避/熔断/超时/幂等）（2h）

**学习内容**：
- 重试与退避：区分可重试（限流/超时/5xx）与不可重试错误，指数退避 + 抖动避免惊群
- 熔断器：连续失败达阈值后短路请求快速失败，半开探测恢复，防止级联雪崩
- 超时分层：连接超时、单次调用超时、整体任务超时分级设置，避免长尾阻塞
- 幂等性：为请求设计幂等键，重试/重复提交不产生重复副作用
- 连接与并发管理：连接复用、并发上限、队列缓冲，配合限流稳定吞吐
- 可观测与告警：统一记录调用指标，异常率/延迟/成本超阈值时告警，闭环治理

**检查点**：
- [ ] 能说清重试、退避、熔断、超时各自解决的问题及配合关系
- [ ] 理解幂等键如何保证重试场景下不产生重复副作用
- [ ] 产出一份 API 最佳实践文档（重试/退避/熔断/超时/幂等/可观测）

## Week 33 (D225-D231)：项目 1 核心 -- 企业知识库 AI 客服

### D225 周一：项目 1 -- 需求 + 全栈架构设计（2h）

**学习内容**：
- 需求拆解：明确企业知识库 AI 客服的核心用户故事（员工自助问答、客户工单分流、知识检索引用、转人工兜底）与非功能需求（并发、延迟、准确率、可观测）
- 角色与边界：区分终端用户、客服坐席、知识管理员、系统管理员四类角色的权限边界与典型操作路径
- 全栈架构分层：前端（对话 + 客服后台）→ API 网关（鉴权/限流/路由）→ 业务层（会话/工单/知识）→ AI 层（RAG + Agent）→ 数据层（关系库 + 向量库 + 缓存）
- 技术选型对齐：复用前阶段技术栈（Next.js + FastAPI + PGVector/Milvus + Redis + MySQL/MongoDB），逐项写清选型理由
- 关键质量目标量化：设定首字延迟、检索召回率、答案忠实度、转人工率等可度量指标作为验收基线
- 架构风险预判：单点瓶颈、LLM 不可用降级、知识更新一致性、敏感信息泄漏四类风险的初步应对方向

**检查点**：
- [ ] 产出一页纸的需求清单，区分功能需求与非功能需求并标注优先级
- [ ] 画出全栈分层架构图（ASCII），标明各层职责与数据流向
- [ ] 列出至少 4 项可量化的质量目标及其目标值

**学习内容**：
- 关系库 Schema 设计：用户、智能体配置、会话、消息、工单、知识文档、知识分块元数据等核心表，确定主外键与索引
- 多库职责划分：关系库存业务实体、向量库存知识嵌入、Redis 存会话态与缓存、MongoDB 存对话日志，明确各库写入与读取路径
- API 规格定义：按 RESTful + WebSocket 规划端点（会话创建、消息发送、流式回复、文档管理、工单流转、转人工），统一请求/响应模型与错误码

**检查点**：
- [ ] 产出核心表的 DB Schema（ER 关系 + 关键索引说明）

### D227 周三：项目 1 -- RAG 知识库 + 向量库（2h）

**学习内容**：
- 知识管道复用：基于 Phase 4 的多格式解析 → 分块 → 嵌入 → 入库流程，落地为项目 1 的知识摄入服务
- 分块策略适配：针对企业知识（FAQ、产品手册、政策文档）选择合适的分块粒度与重叠，兼顾召回与上下文完整
- 混合检索落地：向量检索 + BM25 关键词检索 + Rerank 重排，提升企业术语与专有名词场景的命中率
- 元数据过滤：按部门、文档类型、可见范围等元数据做权限隔离检索，避免越权返回

- 引用可追溯：检索结果携带来源文档与片段定位信息，为前端引用展示与可信度建设打底

**检查点**：

### D228 周四：项目 1 -- Agent 链路（意图识别/工具/转人工）（2h）

**学习内容**：
- 意图识别设计：在入口对用户输入做意图分类（知识问答、业务操作、闲聊、投诉转人工），决定后续走 RAG 还是工具或人工
- Agent 编排：复用 Phase 5 的 ReAct/Tool-Use 模式，把知识检索、工单查询、订单查询等封装为工具供 Agent 调度
- 工具生态规划：定义客服场景必需的工具集（查知识、查工单状态、创建工单、查物流、转人工），明确每个工具的输入输出与权限
- 转人工策略：设定触发转人工的条件（低置信度、负面情绪、显式请求、敏感操作），并设计上下文移交给坐席的数据格式
- 安全护栏：输入过滤（拒绝违规/越权请求）+ 输出过滤（防敏感信息泄漏）+ 最大迭代限制，防止 Agent 失控
- 降级与兜底：LLM 或工具不可用时回退到固定话术或直接转人工，保证对话不中断

**检查点**：
- [ ] 能画出"输入 → 意图识别 → RAG/工具/转人工"的 Agent 决策流程图
- [ ] 定义至少 5 个客服工具的契约（名称/描述/参数/权限）
- [ ] 明确转人工触发条件与上下文移交的数据结构

- 并发与隔离：单用户多会话、坐席同时接多个会话的并发模型与数据隔离
- 消息可靠性：流式中断、网络抖动场景下的消息补偿与幂等，避免重复或丢字

**检查点**：
- [ ] 会话状态机定义完整，含转人工接管与超时回收
- [ ] WebSocket 能稳定推送流式回复与中间状态提示

## Week 34 (D232-D238)：项目 1 完善 + 部署上线

### D232 周一：项目 1 -- 多轮上下文 + 记忆（2h）

**学习内容**：
- 短期记忆：在会话内维护对话历史与中间结果，配合上下文窗口裁剪保证多轮连贯

- 记忆压缩：长对话的摘要与滑动窗口策略，在 Token 预算内保留关键信息
- 上下文注入：检索知识、历史摘要、用户画像如何有序拼装进 Prompt，避免互相干扰与超长
- 记忆隔离：用户私有记忆与企业公共知识的边界，防止串号与隐私泄漏
- 记忆质量验证：多轮指代消解、上下文一致性、跨会话召回准确度的自测方法

**检查点**：
- [ ] 多轮对话中能正确处理指代与上文引用，不丢失上下文
- [ ] 跨会话能召回用户历史偏好或问题摘要
- [ ] 长对话经压缩后仍保留关键信息，未超出上下文窗口

### D233 周二：项目 1 -- 缓存 + 限流 + 降级（2h）

**学习内容**：
- 语义缓存：复用 Phase 4 的 Redis 语义缓存，对高频相似问题命中缓存，降低延迟与成本
- 缓存策略：缓存键设计、TTL、命中率监控、知识更新时的缓存失效处理
- 限流设计：按用户、按接口、按租户的多维限流，区分普通问答与昂贵 Agent 任务
- 熔断与降级：LLM/向量库/工具异常时的熔断阈值与降级链路（缓存回答 → 固定话术 → 转人工）
- 成本控制：Token 预算、单次请求成本上限、月度费用预警，识别高成本调用路径并优化
- 稳定性验证：在依赖故障注入下验证降级链路是否如期生效

**检查点**：
- [ ] 语义缓存对相似问题命中，命中率可监控且达到设定目标
- [ ] 多维限流生效，超额请求被正确拒绝或排队
- [ ] 注入 LLM 故障后系统按降级链路兜底，对话不中断

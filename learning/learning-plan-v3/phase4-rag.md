## Week 10 (D64-D70)：RAG 基础 + 向量检索

### D64 周一：Embedding 原理（向量空间 / 余弦 / 维度 / bge-m3）（2h）

**学习内容**：
- 向量语义表示：文本经嵌入模型映射为高维稠密向量，语义相近的文本在向量空间中距离更近，对照传统关键词匹配的「字面 vs 语义」差异
- 向量空间与维度：维度数（如 384 / 768 / 1024）决定表达能力与存储成本，维度越高语义越精细但检索越慢
- 相似度度量：余弦相似度（关注方向不关注模长）、欧氏距离、点积三者的几何含义与适用场景
- 归一化与点积等价：向量归一化后点积等价于余弦相似度，是向量库默认存归一化向量的原因
- bge-m3 模型特性：支持 100+ 语言、稠密 / 稀疏 / 多向量三种检索粒度、8192 长上下文，适合中英日多语言知识库
- 嵌入的工程约束：同一知识库必须用同一嵌入模型，换模型需全量重新嵌入，模型选型是早期关键决策

**检查点**：
- [ ] 能用自己的话解释「为什么语义相近的句子向量距离更近」
- [ ] 能说清余弦相似度与欧氏距离的差异，以及为何 RAG 常用余弦
- [ ] 理解换嵌入模型必须全量重嵌的工程代价，能说出 bge-m3 的多语言优势

### D66 周三：裸写向量搜索（numpy 余弦相似度）（2h）

**学习内容**：
- 暴力检索原理：将查询向量与库中所有向量逐一计算相似度，取 Top-K，理解其 O(N) 复杂度
- numpy 实现余弦：向量归一化后用矩阵乘法一次性算出查询与全库的相似度，避免 Python 循环
- Top-K 选取：`argsort` / `argpartition` 取最高分的 K 条，理解 partition 比全排序更高效
- 内存与规模瓶颈：暴力检索在万级可接受、百万级不可行，引出近似最近邻（ANN）索引的必要性
- ANN 索引概览：HNSW（图索引）、IVF（倒排聚类）、PQ（乘积量化）如何用「精度换速度」
- 裸写的价值：理解向量检索本质后再用向量库，能看懂索引参数（如 HNSW 的 M / ef）的含义

**检查点**：
- [ ] 能用 numpy 实现一个对全库做余弦相似度并返回 Top-K 的检索函数
- [ ] 能说清暴力检索的复杂度瓶颈，以及 ANN 为何能在精度损失下大幅提速
- [ ] 理解 HNSW / IVF 各自的核心思想（图导航 vs 聚类倒排）

## Week 11 (D71-D77)：分块 + 检索增强

### D71 周一：分块策略（固定 / 递归 / 语义 / 按结构）（2h）

**学习内容**：
- 为何要分块：嵌入与上下文窗口都有长度上限，且过长片段会稀释语义、降低检索精度
- 固定长度分块：按字符 / Token 数硬切，实现简单但易切断句子与语义
- 递归字符分块：按段落 → 句子 → 词的优先级递归切分，尽量在自然边界断开，是通用首选
- 语义分块：用嵌入相似度判断相邻句子是否属同一主题，主题切换处断开，质量高但成本高
- 按结构分块：依据 Markdown 标题、PDF 章节、表格边界等文档结构切分，保留逻辑完整性
- 分块粒度权衡：块过小召回精准但上下文不足，块过大上下文丰富但检索噪声多，需按文档类型调优

**检查点**：
- [ ] 能说清四种分块策略的原理与各自适用的文档类型
- [ ] 能解释「块大小」对检索精度与上下文充分度的双向影响
- [ ] 能为「技术手册 / FAQ / 法律合同」三类文档各选一种合适的分块策略

---

### D72 周二：分块优化（overlap / 元数据 / 父子文档）（2h）

**学习内容**：
- 重叠分块（overlap）：相邻块共享一段重叠内容，避免关键信息恰好被切在边界而丢失
- overlap 比例权衡：重叠越多上下文越连续但冗余存储与召回重复越严重，常取块长的 10%-20%
- 块级元数据增强：为每块附标题路径、所属章节、上下文摘要，提升检索与重排的判别力
- 父子文档检索：用小块（子）做精准检索、命中后返回大块（父）供 LLM 阅读，兼顾精度与上下文
- 句子窗口检索：检索命中单句，但向 LLM 提供该句前后窗口，是父子思想的轻量变体
- 上下文压缩：召回后对片段做相关性裁剪 / 摘要，减少送入 LLM 的无关内容与 Token 成本

**检查点**：
- [ ] 能实现带 overlap 的分块，并说清 overlap 解决的边界丢失问题
- [ ] 能解释父子文档检索如何同时兼顾「检索精准」与「上下文充分」
- [ ] 理解上下文压缩对降低 Token 成本与减少噪声的作用

### D73 周三：BM25 + 混合检索原理（2h）

**学习内容**：
- BM25 原理：基于词频（TF）与逆文档频率（IDF）的稀疏检索，对关键词 / 专有名词 / 代码命中精准
- 稀疏 vs 稠密：BM25 强在精确字面匹配，向量检索强在语义泛化，二者优势互补
- 混合检索动机：纯向量会漏掉「字面精确但语义偏」的查询，纯 BM25 会漏掉「换种说法」的查询
- 分数融合策略：加权求和需归一化不同量纲；RRF（倒数排名融合）只用排名不依赖分数，更稳健
- RRF 机制：对每路检索的排名取倒数加权求和，避免不同检索器分数分布差异带来的失衡
- 权重调参：根据知识库特性（术语密集 vs 口语化）调整稀疏 / 稠密权重，需用评测集验证

**检查点**：
- [ ] 能说清 BM25 与向量检索各自擅长与失效的查询类型
- [ ] 能解释 RRF 为何比「加权分数求和」更稳健（不依赖分数量纲）
- [ ] 理解混合检索权重需结合知识库特性与评测集调优

### D74 周四：Rerank 重排序（CrossEncoder / LLM-based）（2h）

**学习内容**：
- 两阶段检索范式：第一阶段召回（向量 / BM25 取较多候选）+ 第二阶段精排（Rerank 取最终 Top-K）
- 双塔 vs 交叉编码：召回用双塔（查询与文档独立编码、可预存、快）；Rerank 用 CrossEncoder（查询与文档拼接共同编码、精但慢）
- CrossEncoder 重排：对「查询-候选」逐对打分，捕捉细粒度相关性，显著提升 Top-K 精度
- bge-reranker 等模型：多语言重排模型选型，输入候选数与延迟的权衡
- LLM-based 重排：用 LLM 直接对候选打分 / 排序，灵活但成本与延迟更高，适合候选少的高价值场景
- 召回-精排数量配比：召回 N（如 20-50）→ 重排取 K（如 3-5），N 越大召回率越高但重排越慢

**检查点**：
- [ ] 能说清「召回 + 精排」两阶段范式及双塔与交叉编码的分工
- [ ] 能解释 CrossEncoder 为何比双塔更精但不能直接用于全库召回
- [ ] 理解召回数 N 与最终 K 的配比对召回率与延迟的影响

### D75 周五：查询增强（HyDE / 多查询 / 查询改写）（2h）

**学习内容**：
- 查询与文档的语义鸿沟：用户短问与文档长答的表达差异会降低召回，查询增强意在弥合
- HyDE（假设性文档嵌入）：先让 LLM 生成一段「假设答案」，用其向量去检索，使查询更贴近文档表达
- 多查询生成：用 LLM 把一个问题改写成多个不同角度的子查询，分别检索后合并去重，提升召回覆盖
- 查询改写：澄清歧义、补全省略、术语标准化，把口语化问题改写为更利于检索的形式
- 多轮对话查询重构：结合历史对话把「它 / 那个」等指代消解为完整独立查询
- 增强的代价：每次增强都增加 LLM 调用与延迟，需按价值场景选择性启用

**检查点**：
- [ ] 能用自己的话解释 HyDE 为何能缩小「查询-文档」语义鸿沟
- [ ] 能说清多查询生成如何提升召回覆盖率及其去重必要性
- [ ] 理解查询增强带来的额外延迟 / 成本，能判断何时值得启用

## Week 12 (D78-D84)：LangChain / LlamaIndex / Dify 框架

### D78 周一：LangChain 核心（Chain / Prompt / OutputParser / Memory）（2h）

**学习内容**：

### D96 周五：元数据过滤 + 多租户隔离设计（2h）

**学习内容**：

- 前过滤 vs 后过滤：先过滤再检索（精准但可能漏召回）vs 先检索再过滤（召回足但效率低），及各库支持差异

- 性能与隔离权衡：隔离粒度越细越安全但运维 / 资源成本越高，需按租户规模与合规要求取舍

## Week 15 (D99-D105)：进阶 RAG + 评估 + Buffer

### D99 周一：进阶 RAG 模式（CRAG / Self-RAG / Adaptive RAG）（2h）

**学习内容**：
- 朴素 RAG 的缺陷：检索质量差时仍硬答、无相关内容也不拒答、无法自我纠错，进阶模式针对性改进
- CRAG（纠正式 RAG）：用检索评估器给召回打分，质量差时触发查询改写 / Web 检索等纠正动作
- Self-RAG：模型自反思——按需检索、对召回与自身答案做相关性 / 支持度 / 有用性自评并取舍
- Adaptive RAG：按问题复杂度自适应路由——简单问题直答、单跳问题单次检索、多跳问题迭代检索
- 多跳 / 迭代检索：复杂问题分解为子问题逐步检索、用中间结果驱动下一轮检索
- 模式取舍：进阶模式提升鲁棒性但增加 LLM 调用与延迟，需按问题复杂度与质量要求选择性启用

**检查点**：
- [ ] 能说清 CRAG / Self-RAG / Adaptive RAG 各自解决朴素 RAG 的哪类缺陷
- [ ] 能解释 Adaptive RAG 如何按问题复杂度路由检索策略
- [ ] 理解进阶模式在鲁棒性与延迟 / 成本之间的权衡

### D100 周二：多模态 RAG（图片 / 表格 / 代码）（2h）

**学习内容**：
- 多模态 RAG 动机：企业文档含大量图表 / 截图 / 代码，纯文本 RAG 会丢失这类信息
- 图片处理路径：用多模态模型生成图片描述后嵌入文本，或用 CLIP 类模型做图文统一向量空间检索
- 表格处理：表格转 Markdown / 结构化文本、按行嵌入、保留表头语义，避免行列错位丢义
- 代码处理：按函数 / 类等语法单元分块，保留签名与文档串，代码专用嵌入模型提升检索质量
- 统一检索与定位：多模态片段统一进向量库，召回后保留模态类型与定位元数据供前端按类型渲染

### D101 周三：RAGAs 评估（faithfulness / relevancy / precision / recall）（2h）

**学习内容**：
- RAGAs 定位：面向 RAG 的自动化评估框架，无需大量人工标注，用 LLM 充当评判者
- 忠实度（Faithfulness）：答案是否完全基于召回上下文、有无幻觉 / 编造，是 RAG 质量核心指标
- 答案相关性（Answer Relevancy）：答案是否切题、是否回应了用户问题，惩罚跑题与冗余
- 上下文精确率 / 召回率（Context Precision / Recall）：召回上下文中相关占比、相关信息被召回比例，评估检索环节
- 评估数据集：构建「问题 / 标准答案 / 召回上下文」样本集，区分检索指标与生成指标的诊断作用
- 指标驱动诊断：忠实度低查生成 Prompt / 模型，上下文召回低查检索 / 分块，定位问题环节

**检查点**：
- [ ] 能说清 Faithfulness / Answer Relevancy / Context Precision / Recall 各自衡量什么
- [ ] 能区分「检索环节指标」与「生成环节指标」及其诊断指向
- [ ] 能用 RAGAs 在一个评估集上跑出四项指标

### D102 周四：生产 RAG（增量索引 / 文档版本 / 可扩展性）（2h）

## Week 16 (D106-D112)：Agent 基础 + LangGraph

> 本周建立 Agent 心智模型，掌握 LangGraph 这一核心编排框架：从 StateGraph 的节点-边-状态三元素，到状态管理（Reducer / Checkpointer），再到 Tool-Use 执行循环与 Human-in-the-loop 审批，最终两个周末实战产出可运行的 ReAct Agent 与审批流 Agent。

### D106 周一：Agent 范式（ReAct/CoT/Plan-and-Execute/Tool-Use/Autonomous）（2h）

**学习内容**：
- Agent 本质：感知-推理-行动循环（Perceive-Reason-Act），与普通 Chain（固定流程）、单次 Function Calling 的本质区别——Agent 拥有自主决策与多步迭代能力
- ReAct 模式：Reasoning（推理思考）与 Acting（工具行动）交替进行，每步输出 Thought → Action → Observation，适用于需要边想边查的通用任务
- CoT 模式：Chain-of-Thought 思维链，纯推理无外部行动，引导模型显式分步思考，适用于数学、逻辑、规则推演类任务
- Plan-and-Execute 模式：先由 Planner 生成完整计划，再由 Executor 逐步执行，规划与执行解耦，适用于步骤明确的复杂长任务，可降低 Token 与提升可控性
- Tool-Use / Autonomous 模式：Tool-Use 聚焦「何时调用哪个工具」；Autonomous 强调自主目标拆解与自我纠错，开放式探索但风险最高
- 模式选型决策框架：从任务复杂度、可控性要求、延迟容忍度、成本预算四个维度权衡，并理解各模式的失败模式（幻觉、死循环、成本失控）

**检查点**：
- [ ] 能用自己的话讲清 ReAct / CoT / Plan-and-Execute / Tool-Use / Autonomous 五种范式的核心差异
- [ ] 给定三个具体业务场景（如数学题、多步信息检索、可分解长任务），能判断应选哪种范式并说明理由
- [ ] 能说出 Agent 相比 Chain 的优势与三大固有风险，并知道对应的兜底手段

### D107 周二：LangGraph 基础（StateGraph/节点/边/条件路由）（2h）

**学习内容**：
- LangGraph 定位：相比 LangChain 旧版 AgentExecutor 的优势——显式状态管理、图可视化、细粒度流程控制、原生支持循环与中断
- StateGraph 核心模型：图（Graph）= 节点（Node）+ 边（Edge）+ 全局状态（State），理解三元素的关系
- 节点定义：每个节点是一个函数，接收当前 State、返回 State 的局部更新（部分字典）
- 边的类型：普通边（固定流转，`add_edge`）与条件边（基于状态动态路由，`add_conditional_edges`），以及特殊入口 `START` 与终点 `END`
- 条件路由：路由函数读取状态字段、返回下一节点名（或映射键），实现「继续调工具 vs 结束」等分支决策
- 图的编译与执行：`graph.compile()` 生成可运行 Runnable，`invoke`（同步等完整结果）与 `stream`（逐步流式输出）两种调用方式

**检查点**：
- [ ] 能说清 StateGraph 的「节点-边-状态」三元素模型及 START/END 的作用
- [ ] 能画出一个含条件路由的简单 Agent 流程图（LLM 节点 → 路由 → 工具节点 / 结束）
- [ ] 能区分 `invoke` 与 `stream` 的适用场景，知道流式输出对前端体验的价值

### D108 周三：LangGraph 状态管理（TypedDict/Reducer/Checkpointer）（2h）

**学习内容**：
- TypedDict 状态定义：用 Python TypedDict 声明状态结构与字段类型，作为图运行时的数据契约
- Annotated 与 Reducer：`Annotated[list, add_messages]` 或 `operator.add` 指定字段的合并策略，实现消息列表「追加」而非「覆盖」
- 状态更新合并机制：节点只返回局部字段更新，框架按 Reducer 规则自动合并进全局状态——理解默认覆盖与自定义 Reducer 的差异
- Checkpointer 检查点：`MemorySaver`（内存）、`SqliteSaver` / `PostgresSaver`（持久化）保存每步状态快照，是多轮记忆与中断恢复的基础
- thread_id 与会话隔离：通过 config 传入 `thread_id`，同一线程内状态连续、不同线程相互隔离，对应多用户多会话
- 时间旅行（Time Travel）：基于检查点回溯到任意历史状态用于调试、回放与分支重跑；状态设计最佳实践——字段最小化、区分临时态与持久态

**检查点**：
- [ ] 能定义一个含消息列表与中间结果的 TypedDict 状态，并为消息字段配置 add_messages Reducer
- [ ] 能说清为什么消息列表需要追加式 Reducer 而非覆盖
- [ ] 能配置 MemorySaver + thread_id，验证状态在多轮对话间保持且会话间隔离

### D109 周四：Tool-Use Agent（工具定义/Schema/执行循环）（2h）

**学习内容**：
- 工具定义规范：函数签名 + docstring 描述 + JSON Schema 参数声明，三者共同决定 LLM 能否正确选择与填参
- `@tool` 装饰器：LangChain 标准工具定义方式，自动从签名与类型注解提取 Schema；清晰的工具名与描述直接影响选择准确率
- LLM 工具调用流程：LLM 输出 tool_calls 请求 → 框架执行工具 → 结果以 ToolMessage 回传 → LLM 据此决定下一步或给出最终答案
- ToolNode：LangGraph 内置工具执行节点，自动解析 AIMessage 中的 tool_calls、并行执行并回填结果
- 工具绑定：`llm.bind_tools(tools)` 让模型感知可用工具列表，理解绑定后模型输出结构的变化
- 执行循环编排：在「LLM 节点」与「ToolNode」之间用条件边构成 ReAct 循环，直到 LLM 不再请求工具；通过 `recursion_limit` 设最大迭代防死循环

**检查点**：
- [ ] 能定义至少两个工具（如搜索、计算），理解 docstring 与 JSON Schema 描述对选择准确率的影响
- [ ] 能画出并讲清 LLM → ToolNode → LLM 的循环执行流程与终止条件
- [ ] 知道如何用 recursion_limit 设置最大迭代次数防止无限循环

### D110 周五：Human-in-the-loop（中断/审批/interrupt_before）（2h）

**学习内容**：
- 人机协作的必要性：高风险操作（删除、转账、写文件）需人工审批，敏感访问需确认，可纠正 Agent 错误决策
- `interrupt_before` / `interrupt_after`：在指定节点执行前/后暂停图运行，等待外部输入，依赖 Checkpointer 保存暂停现场
- 动态中断 `interrupt()` 与 `Command`：在节点内部主动抛出中断、附带待审批载荷，恢复时通过 `Command(resume=...)` 传回人工决策
- 审批工作流设计：Agent 提出行动方案 → 暂停展示详情 → 人工批准 / 拒绝 / 修改参数 → 从检查点继续执行
- 暂停期间的状态读写：检查当前 State、按需修改字段（如改写工具入参）后再恢复，实现「人在回路」的纠偏
- 典型场景：数据库删除审批、大额交易确认、文件系统写操作确认、低置信度回答转人工

**检查点**：
- [ ] 能说清 interrupt_before 与 Checkpointer 的配合关系（为何中断必须有检查点）
- [ ] 能设计一个「危险操作需审批」的 Agent 流程，标出中断点与恢复路径
- [ ] 理解暂停后如何用 Command(resume) 从检查点恢复，并能在恢复前修改状态

## Week 17 (D113-D119)：Planning + Memory + Tool Use 深入

> 本周深挖 Agent 核心三件套：任务规划（分解 / 计划 / 重规划）、记忆管理（短期 / 长期 / 情景 + 向量存储 + 摘要压缩）、工具工程化（设计模式 + Function Calling 路由 / 并行 / 回填）。周末两战分别产出「规划+记忆 Agent」与「跨会话长期记忆 Agent」。

### D113 周一：任务规划 Planning（任务分解/计划生成/重规划）（2h）

**学习内容**：
- Planning 的价值：把复杂目标拆成可执行步骤，降低单步难度、提升可控性与可观测性
- 任务分解策略：自顶向下分解（目标 → 子目标 → 原子动作）、依赖识别（串行 / 并行）、粒度把控
- 计划生成：用结构化输出（如 JSON 步骤列表）让 LLM 产出可机读计划，区分 Planner 与 Executor 角色
- Plan-and-Execute 架构：Planner 生成计划 → Executor 逐步执行 → 每步结果反馈
- 重规划（Replanning）：执行中遇到偏差 / 失败 / 新信息时，触发 Replanner 修正剩余计划，理解何时重规划、避免反复抖动
- LangGraph 落地：用状态保存 plan / past_steps / current_step，用条件边在「执行下一步 vs 重规划 vs 结束」间路由

**检查点**：
- [ ] 能把一个复杂任务（如「调研某主题并产出报告」）分解为带依赖的步骤列表
- [ ] 能说清 Plan-and-Execute 中 Planner / Executor / Replanner 的职责边界
- [ ] 能设计重规划触发条件，避免计划频繁抖动或陷入循环

### D114 周二：记忆管理（短期/长期/情景 + 向量存储）（2h）

**学习内容**：
- 短期记忆（Short-term）：当前会话上下文，存于 LangGraph 状态 + Checkpointer，会话结束随线程消失
- 长期记忆（Long-term）：跨会话的知识 / 偏好积累，存于向量库或 KV 库，支持语义检索召回
- 情景记忆（Episodic）：过去完整交互案例（成功 / 失败经验），Agent 可参考历史做 few-shot 式决策
- 语义记忆 vs 程序记忆：事实知识（语义）与操作技能 / 规则（程序）的区分，对应不同存取方式
- 向量存储记忆实现：交互摘要 → Embedding → 存入向量库 → 相似检索召回相关记忆注入上下文
- 记忆与 RAG 的关系：长期记忆本质是「以 Agent 自身经历为语料的 RAG」，可复用 Phase 4 的检索栈

**检查点**：
- [ ] 能说清短期 / 长期 / 情景记忆的存储位置与生命周期差异
- [ ] 能设计一个结合短期（状态）与长期（向量库）记忆的 Agent 记忆方案
- [ ] 理解长期记忆与 Phase 4 RAG 在技术栈上的复用关系

### D115 周三：记忆工程（摘要/压缩/遗忘/记忆检索）（2h）

**学习内容**：
- 上下文窗口压力：长对话 Token 膨胀导致成本上升与「中间遗忘」，必须做记忆压缩
- 摘要压缩：滚动摘要（把早期对话压缩为摘要 + 保留近期原文）、分段摘要、增量摘要的取舍
- 滑动窗口与 Token 预算：保留最近 N 轮 / 限定 Token 上限，结合摘要兼顾完整性与成本
- 遗忘机制：基于时效、重要性、访问频率的淘汰策略，避免长期记忆无限膨胀与噪声累积
- 记忆检索工程：写入时的去重 / 打标 / 重要性评分；读取时的语义召回 + 时间衰减 + Top-K 控制
- 记忆注入策略：把召回的记忆放入 System Prompt 还是对话历史，以及避免与当前问题冲突

**检查点**：
- [ ] 能实现一个滚动摘要 + 近期原文保留的短期记忆压缩方案
- [ ] 能说出至少两种遗忘 / 淘汰策略及其适用场景
- [ ] 能设计长期记忆的「写入打分 + 读取召回」流程，控制召回数量与相关性

### D116 周四：工具设计模式（幂等/错误处理/超时/安全边界）（2h）

**学习内容**：
- 幂等性设计：同参数多次调用结果一致、避免重复副作用（结合去重键 / 唯一 ID），尤其针对写操作
- 错误处理规范：工具内部捕获异常、返回结构化错误信息（而非抛栈），让 LLM 能读懂并自纠
- 超时管理：为每个工具设合理执行超时，防止单个慢工具阻塞整个 Agent
- 安全边界：权限控制（读写分离 / 目录白名单）、输入校验（防注入 / 防越权）、输出过滤（防敏感信息泄漏）
- 工具描述优化：清晰的名称、用途、参数说明、返回示例直接决定 LLM 选择与填参准确率
- 工具粒度与组合：细粒度工具灵活但决策复杂，粗粒度工具省心但僵化；用组合工具降低 Agent 决策负担

**检查点**：
- [ ] 能审查现有工具，识别幂等性与安全性问题并给出改造点
- [ ] 能为工具添加超时控制与结构化错误返回
- [ ] 能对比优化前后的工具描述对 LLM 选择准确率的影响

### D117 周五：Function Calling 工程化（工具路由/并行/结果回填）（2h）

**学习内容**：
- Function Calling 与 ReAct 的关系：原生工具调用是 ReAct 行动环节的工程化实现
- 工具路由：工具数量多时的分组 / 命名空间 / 检索式工具选择（先召回相关工具再交给 LLM），避免工具列表过长稀释注意力
- 并行工具调用：一次 LLM 输出多个 tool_calls，ToolNode 并行执行以降延迟，注意彼此无依赖才可并行
- 结果回填：多工具结果按 tool_call_id 对应回填为 ToolMessage，保证模型正确关联请求与结果
- 参数校验与修复：对 LLM 生成的入参做 Schema 校验，失败时回传错误让其修正，必要时做一次自动重试
- 调用控制：单轮工具调用上限、总调用预算、失败降级（跳过 / 兜底回答）

**检查点**：
- [ ] 能设计工具数量较多时的路由 / 检索式选择方案
- [ ] 能说清并行工具调用的前提条件与结果按 id 回填的机制
- [ ] 能实现入参 Schema 校验失败后的回传重试逻辑

### D118 周六：实战 -- 带规划 + 记忆的 Agent（多步任务）（8h）

**学习内容**：
- 构建 Plan-and-Execute Agent：Planner 生成结构化计划、Executor 逐步执行、Replanner 按需修正
- 状态设计：plan（步骤列表）、past_steps（已完成 + 结果）、短期记忆（消息）、中间产物
- 工具与执行：复用 D111 工具集，Executor 每步可调工具，结果反馈给 Planner / Replanner
- 短期记忆压缩：长任务中对历史步骤做滚动摘要，控制上下文 Token
- 重规划落地：某步失败或产出与预期不符时触发重规划，验证不抖动、能收敛
- 端到端验证：输入一个需 4-6 步、含分支的多步任务，观察计划生成 → 执行 → 重规划 → 汇总

## Week 18 (D120-D126)：MCP 工具生态

> 本周专攻 Model Context Protocol：从架构 / 传输 / 生命周期，到用 FastMCP 开发 Server、掌握 Tools / Resources / Prompts 三种能力、在 LangGraph 中集成 MCP 客户端，最终周末产出多能力 MCP Server 与 MCP 驱动的 Agent 助手。

### D120 周一：MCP 架构深入（传输层/协议/生命周期）（2h）

**学习内容**：
- MCP 定位：为 LLM 应用提供标准化的工具与上下文访问协议，让工具「一次开发、处处接入」
- 架构分层：Host（宿主应用）→ Client（协议客户端）→ Server（能力提供方），三者职责分离、解耦演进
- 传输层：stdio（本地子进程、低延迟）、SSE / Streamable HTTP（远程跨网络部署），各自适用场景
- 协议消息：基于 JSON-RPC 2.0，request / response / notification 三类消息
- Server 生命周期：初始化握手（capabilities 能力协商）→ 正常通信 → 优雅关闭
- 安全模型：Server 隔离运行、Host 控制权限授予、最小权限原则；MCP 相比直接函数调用的优势——标准化 / 隔离 / 可发现 / 可复用

### D122 周三：MCP 三种能力（Tools/Resources/Prompts）（2h）

**学习内容**：
- Tools（工具）：模型主动调用、可有副作用、执行操作并返回结果——「模型控制」

- Prompts（提示模板）：预定义交互模板、用户主动选用、标准化特定工作流——「用户控制」
- 三者选择决策：需执行操作用 Tool、需读取数据用 Resource、需引导交互用 Prompt
- 组合模式：Resource 提供上下文 + Tool 执行操作 + Prompt 引导流程，协同完成复杂工作流
- 能力发现：Client 初始化时拉取 Server 完整能力清单，理解三者在控制权上的差异

**检查点**：
- [ ] 给定五个具体需求，能正确判断各自应实现为 Tool / Resource / Prompt
- [ ] 能说清三种能力在控制权上的差异（模型 / 应用 / 用户控制）

### D127 周一：多智能体模式（Supervisor/层级/对等/Swarm）（2h）

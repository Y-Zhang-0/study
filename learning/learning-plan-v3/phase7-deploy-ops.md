## Week 26 (D176-D182)：Docker 容器化

### D176 周一：Docker 基础（镜像/容器/卷/网络）（2h）

**学习内容**：
- Docker 核心定位：基于 Linux 命名空间与 cgroups 的轻量级容器化，与虚拟机在隔离粒度、启动速度、资源开销上的本质差异
- 镜像（Image）与容器（Container）的关系：镜像是只读模板、容器是镜像的可写运行实例，分层（Layer）与写时复制（Copy-on-Write）机制
- 镜像分层与缓存：每条指令生成一层，层可复用与缓存，理解分层对体积与构建速度的影响
- 数据卷（Volume）：命名卷、绑定挂载（bind mount）、tmpfs 三种方式，容器无状态化与数据持久化的分离
- 容器网络：bridge（默认桥接）、host（主机网络）、none、自定义网络，容器间通过服务名/容器名互通
- 常用命令体系：`run`/`ps`/`exec`/`logs`/`images`/`build`/`network`/`volume` 的职责与日常运维场景

**检查点**：
- [ ] 能用自己的话说清镜像、容器、卷、网络四者的关系
- [ ] 理解镜像分层与缓存如何影响构建速度和镜像体积
- [ ] 能用 `exec` 进入运行中的容器排查问题

### D177 周二：Dockerfile（多阶段构建/缓存/体积优化）（2h）

**学习内容**：
- Dockerfile 指令体系：`FROM`/`RUN`/`COPY`/`WORKDIR`/`ENV`/`EXPOSE`/`CMD`/`ENTRYPOINT` 的语义与执行时机
- 多阶段构建（Multi-stage Build）：构建阶段编译、运行阶段只拷产物，剥离编译工具链大幅缩减最终镜像
- 构建缓存利用：指令顺序优化（变动少的层前置）、依赖安装与源码拷贝分离，最大化缓存命中
- 体积优化手段：选择精简基础镜像（slim/alpine/distroless）、合并 RUN 指令减少层数、清理包管理缓存
- `.dockerignore` 的作用：排除无关文件减小构建上下文，避免敏感文件与缓存进入镜像
- `CMD` vs `ENTRYPOINT`：默认命令与固定入口的差异，shell 形式与 exec 形式对信号传递的影响

**检查点**：
- [ ] 能写出一个 Python/Node 应用的多阶段构建 Dockerfile
- [ ] 理解指令顺序如何影响构建缓存命中率
- [ ] 对比优化前后镜像体积，说清缩减来源

### D178 周三：Docker Compose（多服务编排）（2h）

**学习内容**：
- Compose 定位：用声明式 YAML 编排多容器应用，适用于本地开发与单机多服务部署
- 服务（services）定义：镜像/构建、端口映射、环境变量、依赖关系（depends_on）的描述
- 网络与服务发现：Compose 自动创建网络，服务间通过服务名作为主机名互相访问
- 数据持久化：volumes 顶层定义与服务挂载，数据库/向量库数据跨容器重建保留
- 环境配置管理：`.env` 文件、`environment` 与 `env_file`、变量插值，区分不同环境配置
- 健康检查与启动顺序：`healthcheck` 定义就绪探测、`depends_on` 配合条件等待依赖服务真正可用

**检查点**：
- [ ] 能用 Compose 编排至少三个相互依赖的服务
- [ ] 理解服务间通过服务名通信的机制
- [ ] 配置 healthcheck 保证依赖服务就绪后再启动

### D180 周五：镜像优化与安全（非 root/扫描/最小镜像）（2h）

**学习内容**：
- 非 root 运行：创建专用用户、`USER` 指令降权运行，限制容器逃逸与提权风险
- 最小化攻击面：distroless/精简基础镜像、只装运行期必需依赖、移除 shell 与包管理器
- 镜像漏洞扫描：Trivy/Grype 等工具扫描 OS 与依赖层 CVE，纳入构建流程做安全门禁
- 敏感信息防护：避免 secret 进入镜像层、用构建参数与运行时注入、扫描镜像历史泄漏
- 镜像签名与可信：镜像哈希固定（digest 引用）、签名验证、可信镜像源，防供应链篡改
- 资源与权限约束：只读根文件系统、丢弃多余 Linux capability、设置资源限制，纵深防御

**检查点**：
- [ ] 能改造 Dockerfile 以非 root 用户运行容器
- [ ] 用扫描工具发现镜像中的高危 CVE 并修复

## Week 28 (D190-D196)：Kubernetes

### D190 周一：K8s 概念（Pod/Deployment/Service/Ingress/ConfigMap）（2h）

**学习内容**：
- K8s 定位与架构：声明式容器编排，控制平面（API Server/Scheduler/Controller/etcd）与工作节点（kubelet/kube-proxy）
- Pod：最小调度单元，一个或多个共享网络与存储的容器，Pod 的临时性与重建特性
- Deployment：声明式管理 Pod 副本，ReplicaSet 维持期望副本数，滚动更新与回滚能力
- Service：为一组 Pod 提供稳定访问入口与负载均衡，ClusterIP/NodePort/LoadBalancer 类型差异
- Ingress：七层 HTTP 路由入口，按域名/路径路由到不同 Service，TLS 终结与统一接入
- ConfigMap：配置数据与镜像解耦，以环境变量或文件挂载方式注入容器

**检查点**：
- [ ] 能说清 Pod/Deployment/Service/Ingress 的职责与协作关系
- [ ] 理解 Pod 的临时性为什么需要 Deployment 维持副本
- [ ] 区分三种 Service 类型的适用场景

- ConfigMap 用法：键值配置以环境变量、命令行参数或卷文件挂载注入容器
- Secret 用法：敏感信息（密码/Token/证书）独立管理，base64 编码与 ConfigMap 的区别
- Secret 安全性：默认仅 base64 编码非加密，需配合 etcd 静态加密、RBAC 限权、外部密钥管理
- 配置热更新：卷挂载的 ConfigMap/Secret 变更可在 Pod 内更新，环境变量注入则需重建 Pod
- 外部密钥集成：External Secrets/云密钥管理（KMS/Vault）对接，避免明文密钥进集群清单
- 配置与环境分层：不同环境用不同 ConfigMap/Secret，配合命名空间隔离

**检查点**：
- [ ] 能用 ConfigMap 以环境变量与文件两种方式注入配置
- [ ] 用 Secret 管理数据库凭证并挂载到 Pod
- [ ] 说清 Secret 默认非加密的风险与加固手段

### D193 周四：K8s 自动扩缩容（HPA/资源限制）（2h）

**学习内容**：
- 资源请求与限制：requests（调度依据）与 limits（上限约束）对 CPU/内存的设定及其对调度与稳定性的影响
- QoS 等级：Guaranteed/Burstable/BestEffort 三级，资源紧张时的驱逐优先级
- HPA 原理：水平 Pod 自动扩缩，基于 CPU/内存或自定义指标对比目标值动态调整副本数
- 自定义指标扩缩：基于 QPS、队列深度、延迟等业务指标扩缩，需 Metrics Adapter 接入
- 扩缩容行为调优：扩缩容阈值、稳定窗口（避免抖动）、最大最小副本边界
- 与 VPA/Cluster Autoscaler 关系：HPA（横向）、VPA（纵向调资源）、CA（扩节点）的分工配合

**检查点**：
- [ ] 能为工作负载合理设置 requests/limits
- [ ] 配置 HPA 基于 CPU 利用率自动扩缩容
- [ ] 理解扩缩容稳定窗口对避免抖动的作用

### D199 周三：OpenTelemetry 链路追踪（Trace/Jaeger）（2h）

**学习内容**：
- 可观测三支柱：Metrics（指标）、Logs（日志）、Traces（链路）的互补关系与统一价值
- OpenTelemetry 定位：厂商中立的可观测标准，统一 SDK/协议（OTLP）/Collector 采集与导出
- 分布式追踪概念：Trace（一次请求全链路）、Span（单个操作）、上下文传播（TraceID 跨服务透传）

- Jaeger 分析：调用链可视化、依赖拓扑、定位慢 Span 与跨服务延迟瓶颈

**检查点**：

### D200 周四：日志聚合（Loki/ELK/结构化日志）（2h）

**学习内容**：

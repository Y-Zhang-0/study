# 常见坑点与调试技巧

> 转岗路上的避坑指南。分「学习方法」「全栈」「AI/大模型」「Agent」「数据/向量库」「部署运维」「多模态/性能」「求职」八类。

---

## 一、学习方法类

| 坑 | 表现 | 对策 |
|:--|:--|:--|
| 完美主义 | 一个知识点死磕到底，进度停滞 | 卡 30 分钟先跳过，标记 TODO，Buffer 日再补 |
| 收藏家 | 囤教程/视频却不动手 | 每个知识点必须产出可运行代码 |
| 重复造轮子学习 | 把前端、后端、AI 割裂串行 | 坚持融合式：AI 能力一落地就配前端 + 后端 |
| 无复盘 | 学完就忘，无法串成体系 | 每周日做复盘（见 daily-schedule 文末 8 问） |
| 忽视已有优势 | 把 Vue/Node/SQL 当从零学 | 主动迁移：React←Vue、FastAPI←Node、PG←旧 SQL 经验 |

## 二、全栈类

- **React ≠ Vue**：JSX 不是模板；`useEffect` 依赖数组易踩（漏依赖/死循环）；状态不可变更新；列表 `key` 必须稳定。
- **TypeScript 过度类型体操**：先用好接口/泛型/工具类型解决实际问题，别陷入炫技。
- **Next.js 服务端/客户端组件混淆**：`"use client"` 边界、Server Action 与 API Route 的取舍、环境变量泄漏（`NEXT_PUBLIC_` 前缀）。
- **流式渲染卡顿**：逐字渲染未做批处理/节流；Markdown 重渲染未 memo。
- **前后端联调跨域**：CORS 配置、SSE/WebSocket 代理、鉴权 token 传递。

## 三、AI / 大模型类

- **Token 成本失控**：长上下文反复全量传递；未用 Prompt Caching；未设预算上限。**对策**：计数 + 缓存 + 上下文裁剪 + 成本告警。
- **幻觉**：模型编造事实。**对策**：强约束「仅基于提供资料回答」+ 引用来源 + 置信度 + 自检。
- **上下文窗口超限**：长文档/长对话撑爆窗口。**对策**：分块 + 摘要压缩 + 滑动窗口 + 检索召回。
- **结构化输出不稳**：直接让模型「输出 JSON」常失败。**对策**：用 tool_use / function calling 强约束 Schema。
- **流式 + 重试冲突**：流式响应中途失败难重试。**对策**：分清可重试边界，幂等设计。

## 四、Agent 类

- **无限循环**：Agent 反复调用工具不收敛。**对策**：`recursion_limit` / 最大迭代次数 / 终止条件。
- **工具选择错误**：工具描述模糊导致选错。**对策**：清晰的工具名 + docstring + 参数 Schema。
- **多智能体通信混乱**：状态共享/Handoff 设计不清导致死锁或丢上下文。**对策**：明确共享状态字段、隔离私有状态。
- **记忆膨胀**：长期记忆无限增长拖慢检索。**对策**：摘要 + 遗忘策略 + 向量检索 Top-K。
- **框架选型摇摆**：LangGraph/AutoGen/Dify 乱用。**对策**：复杂自定义流程用 LangGraph，标准协作用 AutoGen/CrewAI，低代码编排用 Dify。
- **Agent 不可观测**：出错无从排查。**对策**：LangSmith 全链路追踪 + Token/决策日志。

## 五、数据 / 向量库类

- **向量库选型盲目**：小项目上 Milvus 过重，大规模用 Chroma 扛不住。**对策**：原型用 Chroma/PGVector，规模化用 Milvus；按数据量与过滤需求选。
- **检索质量差**：只用向量检索召回不全。**对策**：混合检索（向量 + BM25）+ Rerank + 查询改写。
- **分块不当**：块太大噪声多、太小丢上下文。**对策**：按结构/语义分块 + overlap + 父子文档。
- **数据库 N+1 / 慢查询**：ORM 懒加载导致大量查询。**对策**：`EXPLAIN` 分析 + 复合索引 + 预加载。
- **缓存三大问题**：穿透/击穿/雪崩。**对策**：空值缓存 + 互斥锁 + 随机过期 + 限流。

## 六、部署运维类

- **镜像臃肿**：单阶段构建、装全量依赖。**对策**：多阶段构建 + 最小基础镜像 + `.dockerignore`。
- **容器内跑 root**：安全风险。**对策**：非 root 用户 + 镜像扫描。
- **有状态服务上 K8s**：本地会话/文件导致扩容失败。**对策**：会话存 Redis、文件存对象存储，服务无状态化。
- **无监控盲飞**：线上出问题靠猜。**对策**：Prometheus + Grafana（Token 成本/延迟 P95/命中率）+ OTel 链路追踪。
- **SLA 无定义**：没有可衡量目标。**对策**：先定 SLI/SLO，再配告警规则。

## 七、多模态 / 性能类

- **GPU 显存爆**：模型 + KV Cache 超显存。**对策**：量化（GPTQ/AWQ）+ 显存估算 + 选合适尺寸模型。
- **推理吞吐低**：逐请求串行。**对策**：vLLM 连续批处理（PagedAttention）+ 并发队列。
- **语音链路延迟高**：整段 ASR→LLM→TTS 串行。**对策**：流式 ASR + 流式 TTS + VAD 打断 + 首字优化。
- **云 GPU 烧钱**：忘记关实例。**对策**：按时租用、用完即停、本地小模型先验证。
- **压测无基准**：调优无依据。**对策**：先建基准（吞吐/延迟/资源），再对比优化前后。

## 八、求职类

- **作品集像玩具**：Demo 跑不起来、无部署链接、README 简陋。**对策**：真上线 + README + 演示视频 + 量化成果。
- **只会用不懂原理**：面试深挖即露馅。**对策**：每个技术能讲「为什么这样设计、有何取舍」。
- **叙事平淡**：罗列「我学了 X」。**对策**：用「问题 → 方案 → 量化效果」讲故事（见 overview 面试叙事）。
- **系统设计慌**：无方法论。**对策**：固定框架——需求 → 估算 → API → Schema → 架构 → 取舍。

## 通用调试心法

1. **最小复现**：把问题缩小到最小可复现单元，再定位。
2. **二分排查**：注释/隔离一半，缩小范围。
3. **读错误信息**：80% 的答案在报错栈里，先认真读完再搜。
4. **打印 + 观测**：前端用 console + DevTools，后端用结构化日志，Agent 用 LangSmith。
5. **先问机制再问 Bug**：理解组件如何工作，往往比直接修更快。

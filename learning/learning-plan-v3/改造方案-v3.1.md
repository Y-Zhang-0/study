# V3 学习计划改造方案(v3.1)

> 立于 2026-06-04(D6 收口后)。起因:D6 暴露实战日强度不足、教学深度不够。
> 目标:把计划从"广度覆盖"升级为"能学透、能转岗"的高强度体系。
> 节奏:**定原则(本文档) + 近期细排(D7/Phase2) + 远期滚动(Phase3-9 走到前一周再细化)**。
> 主公拍板:认可此节奏;先落原则 + 重排 D7 做样板。

---

## 一、诊断

**优点**:覆盖系统、项目驱动、里程碑清晰、边投边学务实——骨架是好的,不推倒。

**风险**(D6 暴露 + 通盘审视):
1. **颗粒度不均**:简单主题(CLI)8h 偏大;难主题(Embedding/LangGraph/RAG 调优)2h 偏小。
2. **深度 vs 广度**:38 周铺太广,深度恐不到"企业级"面试/实战要求。
3. **基础内功缺口**:OS/网络/内存/并发未系统补,而企业级岗位面试必深挖系统设计与原理。
4. **项目区分度**:偏功能堆叠,需加真实复杂度 + 量化成果,简历才打得动。
5. **助教执行**:此前教学"够用即止",强度/深度/资料未拉满(已纠正,本方案约束)。

**转岗判断(不粉饰)**:计划是**必要非充分条件**。严格执行能给扎实广度、达中级 AI 应用工程师知识面;但"企业级全栈"高标准,靠 38 周自学深度有风险,现实路径更可能是"先转中级 AI 应用岗 → 工作中长到企业级"。决定成败的不是覆盖面(已够),是**执行深度 + 内功 + 项目区分度 + 面试**。主公的多年工程经验是真优势,方向没错。

## 二、改造原则(五条,全程适用)

| # | 原则 | 改造动作 |
|:--|:--|:--|
| 1 | **强度** | 实战日 = 真项目 + 扩展挑战 + 深挖(见 §三模板) |
| 2 | **深度** | L2 七段讲到位 + 每点配资料 + 关键机制下探源码/原理 |
| 3 | **内功** | 关联式补 OS/网络/内存/并发(见 §四) |
| 4 | **颗粒度** | 按难度重排 timebox:简单的压缩、难的扩展或拆分 |
| 5 | **项目区分度** | 真实复杂度 + 量化成果(QPS/延迟/覆盖率/成本),简历能打 |

## 三、实战日新模板(8h,五段)

```
├─ 核心实现(3h)    主线项目,主公亲手写
├─ 扩展挑战(2h)    真实工程特性:重试退避/限流/配置化/测试/日志/可观测
├─ 深度追问(1.5h)  L2 深挖 + 源码/原理下探 + 臣的连环追问
├─ 内功关联(1h)    本主题相关的 OS/网络/内存/并发原理就地补
└─ 资料精读(0.5h)  官方文档关键章节 + 复盘
```

### 2h 工作日模板("窄而深",同样不许"表面应用")

主题窄(一个核心概念/技术),但**深度打满**——2h 也要"学透",不是"跑通 demo 就收"。

```
├─ 核心理解(0.75h)   亲手写最小可运行 + 讲清"它是什么 / 解决什么问题"
├─ 深度追问(0.5h)    L2:底层机制 + 设计动机·取舍 + 边界陷阱(写代码触发并观察)
├─ 内功/对照(0.25h)  关联内功 或 旧栈对照(Java/Node/Vue → Python/TS)
├─ 资料精读(0.25h)   官方文档关键节 + 一个"为什么"查证到底
└─ 考核留痕(0.25h)   自测"能否复述 + 能否迁移",记台账薄弱点
```

**工作日"学透"硬标准(每天至少做到,否则视同未达标)**:
1. **不止"会调"**:每个 API/概念答得出"为什么这样设计 / 底层怎么实现 / 何时不该用"。
2. 至少一处**机制下探**到原理/源码层。
3. 至少一个**边界陷阱亲手验证**(写代码触发,看实际行为,而非听臣说)。
4. 至少一处**旧栈对照或内功关联**,与已有经验打通。
5. **考核留痕**:能复述 + 能迁移才算过,记台账滚动。

> 总纲:**实战日"宽而深"、工作日"窄而深",共同点是「深」——任何一天都不许停在"会用 API"的表层。**

## 六、D7 重排样板(新模板首个样板)

**主题**:给 D6 的 fetcher CLI 补完整测试套件 + 工程化收尾 + 真实健壮性
**产出**:`tests/test_cli.py` 等 + 重试退避 + 信号量限流 + README,覆盖率 >80%

### 核心实现(3h)
- **单元测试**:mock httpx 测 `fetch_posts`/`fetch_in_batch`/`batch_iter`(复用 D4 的 patch/parametrize/asyncio)
- **集成测试**:用 Click 的 **`CliRunner`** 测 `fetch`/`batch` 命令入口(不真跑进程、不打网络)
- **异步路径**:`pytest-asyncio` 覆盖正常 + 异常(超时/404/全失败)
- **覆盖率**:`pytest-cov` 冲 >80%,识别未覆盖分支

### 扩展挑战(2h)——真实工程特性(加强度的核心)
- **重试退避**:给 `fetch` 加**指数退避 + 抖动**(429/5xx/超时才重试,4xx 不重试),并写测试验证重试次数
- **信号量限流**:给批量抓取加 `asyncio.Semaphore`,对照 D6 的 `batch_iter` 分批——**两种限流的取舍**(分批=批间串行;信号量=全程并发但限上限)
- **退出码测试**:`CliRunner` 验证 `result.exit_code`(成功 0 / 参数错 2 / 业务失败非 0)
- **参数化边界**:空输入、全失败、重试耗尽、超时

### 深度追问(1.5h)——L2 深挖
- mock `AsyncClient` 为何要 patch 在**被使用处**(D4 patch 路径的延伸 + 异步 mock 的 `AsyncMock`)
- `CliRunner` 怎么隔离 CLI(`isolated_filesystem`、捕获 stdout/exit_code、不起子进程)
- 覆盖率的**意义与局限**:行覆盖 ≠ 路径覆盖,80% ≠ 没 bug
- `pytest-asyncio` 的 event_loop fixture 机制

### 内功关联(1h)——网络 + 软件工程
- **重试退避背后的网络原理**:为什么要**指数退避 + 抖动**——惊群(thundering herd)、网络拥塞控制、雪崩;固定间隔重试为何放大故障
- **幂等性**:重试的前提(GET 幂等可重试,非幂等需幂等键)
- **测试替身分类**:mock/stub/spy/fake/dummy 的区别(软件工程概念)

### 资料精读(0.5h)
- pytest 官方:fixtures、monkeypatch、`pytest-asyncio`
- Click 官方:Testing(CliRunner)
- 经典:exponential backoff and jitter(AWS Architecture Blog 经典文)

### 验收标准(升级版)
- [ ] 单元 + 集成两层,全绿、离线可跑、覆盖率 >80%
- [ ] 重试退避有测试佐证(重试次数/退避时长可验)
- [ ] 信号量限流落地,能口述 vs 分批的取舍
- [ ] 退出码经 `CliRunner` 验证
- [ ] README + pyproject + ruff,他人可一键安装运行

### 兼本周 Buffer 确认
台账"Buffer 日 D7 确认即划"的薄弱点一并清账:Field 语义、闭包 cell、计时装饰器默写、gather 保序、side_effect、引用复制、patch 路径(连续确认)。

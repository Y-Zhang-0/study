# Phase 6：后端架构 + 多数据库

> **时间**：Week 22-25，D148-D175，104 小时
> **前提**：Phase 5 多智能体系统 + 配置后台完成
> **目标**：构建可扛高并发的 Agent API 网关与多数据库存储层——掌握 Agent API 网关（路由/鉴权/限流/计费）、WebSocket 会话管理、高并发、MySQL/PostgreSQL 业务库设计、MongoDB 对话日志、Redis 会话与分布式锁
> **融合设计**：Agent 应用的真实瓶颈在后端——长连接对话需要 WebSocket + 会话管理，多用户需要鉴权与限流计费，海量对话需要多数据库分工存储，高并发需要连接池/熔断/限流。本阶段以「为 Agent 后端建一座扛得住的网关与存储层」为主线，用问题驱动后端架构学习

## Week 22 (D148-D154)：后端服务架构 + 异步

### D148 周一：FastAPI 进阶（2h）

**学习内容**：
- 中间件机制：`@app.middleware("http")` 与 ASGI 中间件，请求/响应拦截、统一日志、耗时统计、CORS 处理
- 依赖注入（Depends）：函数级依赖、类依赖、子依赖嵌套，用于鉴权校验、数据库会话获取、参数复用
- 依赖的作用域与缓存：同一请求内依赖结果缓存、`use_cache=False` 强制重算、`yield` 依赖实现资源获取与释放
- 后台任务（BackgroundTasks）：响应返回后执行轻量异步任务（日志落库、通知推送），与 Celery 重任务的边界划分
- 生命周期（lifespan）：`asynccontextmanager` 管理应用启停，启动时建连接池/加载模型、关闭时优雅释放资源
- 全局异常处理与依赖覆盖：`exception_handler` 统一错误响应格式，`dependency_overrides` 在测试中替换真实依赖

**检查点**：
- [ ] 能说清中间件、依赖注入、后台任务三者的执行时机差异
- [ ] 实现一个 `yield` 依赖完成数据库会话的获取与自动释放
- [ ] 理解 lifespan 相比已废弃的 `@app.on_event` 的优势

### D149 周二：WebSocket 后端（2h）

**学习内容**：
- WebSocket 协议本质：基于 HTTP 升级（101 Switching Protocols）的全双工长连接，与 SSE/轮询的对比与选型
- FastAPI WebSocket 端点：`@app.websocket`、`accept` / `receive_text` / `send_text` / `close` 的完整握手与收发流程
- 连接管理器（ConnectionManager）：维护活跃连接集合，支持单发、按用户发、按会话广播
- 广播与房间（Room）：将连接按会话/群组分组，实现定向广播，避免全量推送
- 异常与断线处理：`WebSocketDisconnect` 捕获、心跳保活（ping/pong）、超时清理、重连协议设计
- 流式对话集成：Agent 逐 token 输出通过 WebSocket 实时推送，背压（backpressure）与发送队列处理

**检查点**：
- [ ] 能解释 WebSocket 握手过程与 HTTP 升级机制
- [ ] 实现一个 ConnectionManager 支持单发与按会话广播
- [ ] 理解断线检测（心跳）与连接清理对长连接服务的必要性

### D150 周三：会话管理（2h）

**学习内容**：
- 会话（Session）模型：会话标识、归属用户、智能体绑定、创建/更新时间、消息序列的数据结构设计
- 会话存储选型：内存（单机临时）vs Redis（分布式共享）vs 数据库（持久化），各自的适用场景与组合策略
- 上下文窗口管理：消息历史的 Token 计数、超窗裁剪策略（滑动窗口/摘要压缩/重要性保留）
- 多轮状态维护：在无状态 HTTP/WebSocket 之上重建会话状态，区分临时态与持久态
- 会话生命周期：创建、激活、过期（TTL）、归档、删除的完整流转与清理机制
- 并发会话隔离：同一用户多设备/多标签页的会话隔离，会话锁防止并发写冲突

**检查点**：
- [ ] 能设计一个支持多轮上下文的会话数据结构

### D151 周四：Agent API 网关设计（2h）

**学习内容**：
- 网关定位：作为 Agent 后端的统一入口，承担路由、鉴权、限流、计费、可观测的横切职责，与业务逻辑解耦
- 路由与转发：按路径/版本/智能体类型路由到不同后端服务，支持灰度与多版本共存
- 鉴权方案：API Key（服务间）、JWT（用户态）、OAuth2 的对比，鉴权信息透传与上下文注入
- 限流维度：按用户/按 API Key/按智能体/全局多维限流，配额（Quota）与速率（Rate）的区别
- 计费模型：按请求数/按 Token 消耗/按时长计费，用量统计的采集点与聚合方式
- 网关分层架构：接入层（鉴权限流）→ 路由层（转发）→ 业务层（Agent 执行）→ 数据层，各层职责边界

**检查点**：
- [ ] 能画出 Agent API 网关的分层架构图与请求流转路径
- [ ] 对比 API Key 与 JWT 鉴权的适用场景
- [ ] 设计一个按 Token 消耗计费的用量采集方案

### D152 周五：Celery 异步任务（2h）

**学习内容**：
- Celery 三要素回顾与定位：Worker（执行）、Broker（消息队列 Redis/RabbitMQ）、Backend（结果存储）在 Agent 后端中的角色
- 任务定义与调用：`@app.task` 装饰器、`delay()` / `apply_async()`、指定队列与优先级路由
- 任务模式：重试（指数退避）、Chain（串行）、Group（并行）、Chord（并行后汇总）在 Agent 流水线中的应用
- 与 FastAPI 集成：API 提交任务返回 task_id，客户端轮询或 SSE 推送进度，`update_state` 上报中间进度
- Worker 管理与并发：并发模型（prefork/gevent/eventlet）选择、prefetch 配置、对 IO 密集型 LLM 调用的适配
- 长任务场景：Agent 复杂推理耗时长，异步化避免 HTTP 超时，结果落库后通过 task_id 关联查询

**检查点**：
- [ ] 能说清 Broker 与 Backend 的角色差异
- [ ] 实现一个带指数退避重试的异步任务
- [ ] 理解 LLM IO 密集场景下 Worker 并发模型的选择

### D153 周六：实战 -- Agent API 网关（鉴权+限流+流式）（8h）

**学习内容**：
- 搭建 Agent API 网关骨架：FastAPI 应用 + 中间件链 + 依赖注入鉴权 + lifespan 管理连接池
- 鉴权实现：API Key 校验中间件 + JWT 用户态解析，鉴权失败统一返回 401/403
- 限流实现：基于 Redis 的滑动窗口/令牌桶限流，按用户与 API Key 多维度限速，超限返回 429
- 计费埋点：在网关层采集请求数与 Token 消耗，异步写入用量记录，支持配额校验
- 流式转发：网关将后端 Agent 的流式输出（SSE/逐 token）透传给客户端，保持背压与连接稳定
- 端到端联调：客户端带 Key 请求 → 鉴权 → 限流校验 → 路由到 Agent → 流式返回 → 用量记账

**检查点**：
- [ ] 无效/缺失凭证被网关拦截，合法请求正确路由到后端
- [ ] 限流生效：超过速率阈值的请求返回 429，恢复后可正常访问
- [ ] 流式响应端到端透传，首字延迟与连接稳定性达标
- [ ] 每次请求的用量（请求数/Token）被正确采集并可查询

## Week 23 (D155-D161)：关系型数据库（MySQL / PostgreSQL）

### D155 周一：MySQL vs PostgreSQL 选型 + SQL 回顾（2h）

**学习内容**：
- 两者定位差异：MySQL（生态成熟、读多写少、互联网主流）vs PostgreSQL（功能强大、复杂查询、JSON/向量扩展、标准 SQL 严格）
- 数据类型对比：PostgreSQL 的 JSONB、数组、枚举、UUID、范围类型；MySQL 的 JSON、ENUM 支持
- 扩展能力：PostgreSQL 的 PGVector（向量检索，与 Phase 4 呼应）、全文检索、扩展生态 vs MySQL 的存储引擎（InnoDB）
- 并发与事务：MVCC 实现差异、隔离级别默认值、锁机制粒度的对比
- SQL 回顾：JOIN（内/外/自连接）、子查询、聚合与 GROUP BY、窗口函数、CTE（WITH 子句）
- Agent 场景选型：业务库（用户/会话/消息）的选型考量，为什么本计划倾向 PostgreSQL（JSONB 存元数据 + 向量扩展统一栈）

**检查点**：
- [ ] 能列出 MySQL 与 PostgreSQL 各自的三个核心优势
- [ ] 熟练写出含 JOIN、窗口函数、CTE 的复杂查询

### D159 周五：事务与并发（隔离级别/锁/乐观悲观锁）（2h）

### D163 周二：MongoDB 建模（嵌入 vs 引用/索引/分片）（2h）

**学习内容**：
- 嵌入（Embedding）vs 引用（Reference）：一对少嵌入、一对多/多对多引用，读写模式驱动建模决策
- 反范式设计：MongoDB 倾向反范式以减少 JOIN，用冗余换读性能，与关系库范式化对比
- 文档大小限制：单文档 16MB 上限，无限增长数组（unbounded array）反模式与规避
- 索引策略：复合索引最左前缀、覆盖索引、索引选择性，针对查询模式建索引
- 分片（Sharding）：分片键选择、范围分片 vs 哈希分片、数据分布均衡，水平扩展原理
- 副本集（Replica Set）：主从复制、读写分离、故障自动切换（failover）

**检查点**：

### D165 周四：Redis 进阶（发布订阅/Stream/Lua/持久化）（2h）

**学习内容**：
- 发布订阅（Pub/Sub）：频道订阅与消息广播，适用于实时通知，及其消息不持久化的局限
- Stream 数据结构：消费者组（Consumer Group）、消息确认（ACK）、阻塞读取，作为轻量消息队列
- Lua 脚本：原子性执行多条命令，避免竞态（如限流计数+判断、库存扣减的原子操作）
- 持久化机制：RDB（快照，恢复快但可能丢数据）vs AOF（追加日志，更可靠但体积大），混合持久化
- 数据结构进阶：Sorted Set（排行榜/延迟队列）、HyperLogLog（基数统计）、Bitmap（活跃统计）
- 高可用：主从复制、哨兵（Sentinel）故障转移、Cluster 集群分片

**检查点**：
- [ ] 能区分 Pub/Sub 与 Stream 的可靠性差异及适用场景

## Week 25 (D169-D175)：高并发 + 架构 + Buffer

### D169 周一：高并发理论（C10K/事件循环/IO 模型）（2h）

**学习内容**：
- C10K 问题：单机万级并发连接的挑战，从「一连接一线程」到事件驱动的演进
- IO 模型：阻塞 IO、非阻塞 IO、IO 多路复用（select/poll/epoll）、异步 IO，各自的原理与代价
- 事件循环：epoll/kqueue 驱动的事件循环机制，单线程处理海量连接的原理（Nginx/Node/asyncio）
- 并发模型对比：多进程、多线程、协程（goroutine/asyncio）、事件驱动的取舍
- CPU 密集 vs IO 密集：LLM 调用为 IO 密集，适合协程/事件驱动；区分两类负载的扩展策略
- 性能瓶颈维度：连接数、吞吐、延迟、资源占用，定位瓶颈的方法论

**检查点**：
- [ ] 能解释 C10K 问题及事件驱动如何解决

### D170 周二：连接池与资源复用（DB/HTTP/Redis 连接池）（2h）

**学习内容**：

- 数据库连接池：池大小（最小/最大）、空闲超时、连接存活检测、SQLAlchemy/asyncpg 池配置
- HTTP 连接池：Keep-Alive 复用、最大空闲连接、每主机连接数、连接复用对 LLM API 调用的收益
- Redis 连接池：连接数配置、连接复用、与多实例共享的协调
- 池大小调优：过小导致排队、过大耗尽资源，结合并发量与后端能力设定
- 资源泄漏防护：连接未释放、池耗尽的监控与告警，`yield` 依赖/context manager 保证释放

**检查点**：

### D171 周三：限流与熔断（令牌桶/滑动窗口/熔断器）（2h）

**学习内容**：
- 限流算法：固定窗口（边界突刺问题）、滑动窗口（更平滑）、漏桶（恒定速率）、令牌桶（允许突发）
- 限流维度与实现：单机限流 vs 分布式限流（Redis+Lua），按用户/API Key/全局多维度
- 熔断器原理：闭合（正常）/打开（快速失败）/半开（试探恢复）三态机，错误率阈值触发
- 熔断应用：下游 LLM API 故障时快速失败避免雪崩，半开探测恢复后自动闭合
- 降级策略：限流/熔断触发后的降级（缓存回答、固定话术、排队、转人工），保证核心可用
- 与网关集成：限流前置拦截、熔断包裹下游调用，呼应 Week 22/25 的网关设计

**检查点**：
- [ ] 能对比四种限流算法的特性与适用场景
- [ ] 理解熔断器三态转换的触发条件

### D172 周四：分布式锁与一致性（Redis 锁/Redlock/幂等）（2h）

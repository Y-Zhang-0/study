# 实战项目：开源参考 + 落地思路

> 用法：**开源项目用来"读"（学架构与模式），自己的项目用来"建"（进作品集）**。
> 切忌 clone 一个开源项目改改就当作品——面试一深挖即露馅。正确姿势：读开源项目学它怎么设计，再用自己的理解从 0 搭一个**能讲清每一处取舍**的项目。
> ⚠️ 知识截至 2026-01，下列开源项目以**名称/组织**为准，star 数与活跃度临用时核一下。

## 二、实战项目思路（建，进作品集）

> 与 V3「贯穿主线」对齐：每阶段产出一个，逐阶段长大，**始终保持线上部署**。

### 主线项目（跟随阶段自然长大）
| 阶段 | 项目 | 关键卖点（面试可讲） |
|:--|:--|:--|
| P2 | 多厂商 LLM 客户端库 + Prompt 工作台 | 统一抽象、流式/重试/降级、Token 成本对比 |
| P3 | AI 对话 Web 应用（Chatbot） | 流式逐字、多会话、配置后台、上传界面 |
| P4 | 企业知识库 RAG 问答（全栈上线） | 多格式解析、混合检索+Rerank、Redis 缓存命中率、引用溯源 |
| P5 | 多智能体系统 + 配置后台 | ReAct/Planning/Memory、MCP 工具、Supervisor 协作、LangSmith 观测 |
| P6-P7 | 升级为高并发 + 云原生 | WebSocket 会话、限流熔断、K8s/HPA、监控告警 SLA |
| P8 | 多模态升级 | 语音对话、图像生成插件 |

### 旗舰作品（P9，2 选 2 或择优）
- **企业知识库 AI 客服**：意图识别 → RAG 检索 → 工具调用 → 转人工兜底；多轮记忆、并发、监控、上线。**最贴合 JD 的"AI 客服"案例。**
- **AI 自动化工具**（任选其一，越垂直越亮眼）：
  - 简历优化 / JD 匹配 Agent（你正在求职，自用又能演示）
  - 文档批处理 pipeline（上传 → 解析 → 抽取 → 结构化 → 入库，Agent 编排）
  - 自动化运营助手（定时任务 + 工具调用 + 报告生成）
  - 代码仓库问答 / Code Review Agent

### 🎯 差异化亮点（用你的工业/AGV 背景，作品集脱颖而出）
> 大多数转岗者做的是"通用客服/通用知识库"，同质化严重。**你的工业领域经验是别人没有的护城河**：
- **工业设备知识库 AI 客服**：设备手册/工单/故障库 RAG + 故障诊断 Agent
- **AGV/产线运维助手**：日志/告警接入 + 根因分析 Agent + 处置建议
- **工单自动化 Agent**：自然语言 → 结构化工单 → 派发 → 跟踪
这类项目**懂业务 + 懂 AI**，面试叙事天然有说服力（"我在工业场景落地过 AI"）。

## 三、什么样的作品集才"加分"

1. **真上线**：有可访问的部署链接（哪怕小流量），不是只跑在本地的 Demo。
2. **README 讲故事**：问题背景 → 架构图 → 技术选型理由 → 关键难点 → **量化成果**（延迟/命中率/成本）。
3. **演示视频**：1-3 分钟录屏，HR/面试官没时间跑你的代码。
4. **能深挖**：每个技术点都能回答"为什么这么设计、还有什么方案、瓶颈在哪"。
5. **有 eval**：RAG 有 RAGAs 分数、Agent 有任务完成率——体现工程严谨。
6. **代码质量**：测试、CI、清晰结构、commit 规范——体现工程素养。

> 宁可**2 个深、能讲透的项目**，也不要 5 个浅、说不清的 Demo。

## 四、避免的坑
- ❌ Clone 开源项目改改就当原创——一问设计细节就露馅
- ❌ 追求功能多而杂——不如一个核心闭环打磨到生产级
- ❌ 只做后端 API、没有 UI——JD 要全栈，作品要"看得见"
- ❌ 做完不部署、不写 README——等于没做
- ✅ 读开源学模式 → 自己从 0 建 → 上线 → 量化 → 写复盘 → 讲故事

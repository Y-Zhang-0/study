## Week 6 (D36-D42)：Next.js 基础

### D36 周一：Next.js App Router（路由/布局/RSC）（2h）

**学习内容**：
- App Router 目录约定：`app/` 下文件夹即路由段，`page.tsx` 定义页面、`layout.tsx` 定义共享布局、`loading.tsx` 与 `error.tsx` 处理加载与错误边界
- React Server Components（RSC）默认机制：组件默认在服务端渲染，不打包进客户端 bundle，需交互时用 `"use client"` 显式标记客户端组件
- 服务端组件与客户端组件的边界：数据获取与敏感逻辑留在 RSC，事件处理与浏览器 API 留在客户端组件，二者通过 props 单向传递
- 嵌套布局与模板：`layout` 跨路由持久不重渲染，`template` 每次导航重建，适用于需要重置状态的场景
- 动态路由与路由组：`[id]` 动态段、`[...slug]` 捕获所有、`(group)` 路由组不影响 URL 仅组织代码
- 导航 API：`<Link>` 客户端预取跳转、`useRouter` 编程式导航、`redirect` 服务端重定向
- 产出：搭出一个含嵌套布局与动态路由的 Next.js 路由示例

**检查点**：
- [ ] 能说清 RSC 与客户端组件的渲染位置差异，以及何时必须加 `"use client"`
- [ ] 能用 `app/` 目录约定搭出含共享布局的多页面路由
- [ ] 理解 `<Link>` 预取相比传统 `<a>` 跳转的体验优势

### D37 周二：Next.js 数据获取（Server Actions/fetch/缓存）（2h）

**学习内容**：
- Server Components 中直接 `async/await fetch`：在服务端发起请求，结果直接渲染，无需客户端 loading 态
- fetch 缓存语义：`cache: "force-cache"`（默认静态缓存）、`no-store`（每次请求）、`next: { revalidate }`（定时重新验证）
- Server Actions 定义：`"use server"` 标记的异步函数，可在表单 `action` 或客户端事件中直接调用，无需手写 API 层
- 数据变更与重新验证：`revalidatePath` 与 `revalidateTag` 在变更后精确失效缓存，触发界面刷新
- 渲染策略选择：静态渲染（SSG）、动态渲染（SSR）、增量静态再生（ISR）的适用场景与权衡
- 与 Phase 2 后端的衔接思路：哪些数据走 RSC 直取、哪些交互走 Server Action、哪些实时流走客户端
- 产出：含 fetch 缓存策略与 Server Action 变更的数据获取示例

**检查点**：
- [ ] 能在 Server Component 中直接获取数据并控制其缓存行为
- [ ] 能写一个 Server Action 完成数据变更并触发对应路径重新验证
- [ ] 能判断一个页面应选静态、动态还是 ISR 渲染

### D39 周四：Next.js API Routes + 中间件 + 环境变量（2h）

**学习内容**：
- Route Handlers：`app/api/.../route.ts` 中导出 `GET`/`POST` 等函数，基于 Web 标准 `Request`/`Response` 对象
- 何时用 Route Handler vs Server Action：对外暴露 REST 接口、给第三方/移动端调用用前者，内部表单变更用后者
- 中间件 `middleware.ts`：在请求到达路由前拦截，做鉴权、重定向、请求头改写、地域路由
- 环境变量约定：`NEXT_PUBLIC_` 前缀变量注入客户端，无前缀仅服务端可见，密钥（LLM API Key）严禁暴露给浏览器
- 后端代理模式：前端经 Route Handler 转发到 Phase 2 LLM 后端，隐藏密钥并统一鉴权与限流
- 请求校验与错误响应：入参校验、统一错误结构、HTTP 状态码语义
- 产出：实现一个代理 Phase 2 后端的 API Route 示例

**检查点**：
- [ ] 能写一个 Route Handler 接收请求并代理到后端服务
- [ ] 理解 `NEXT_PUBLIC_` 前缀的暴露边界，能正确隔离 LLM 密钥
- [ ] 能用 middleware 实现一个简单的鉴权拦截

### D40 周五：流式 UI 基础（SSE/ReadableStream/Suspense）（2h）

**学习内容**：
- 流式响应动机：LLM 生成耗时长，逐字返回可大幅降低「首字延迟」感知，提升交互体验
- SSE（Server-Sent Events）协议：基于 HTTP 的单向服务端推流，`text/event-stream` 格式，相比 WebSocket 更轻量
- ReadableStream 与 Web Streams API：服务端构造可读流逐块写出，客户端用 reader 逐块消费
- 前端消费流：`fetch` 响应体的 `body.getReader()`、`TextDecoder` 解码分片、增量拼接渲染
- React Suspense 与流式 SSR：`<Suspense>` 边界配合服务端组件分块流式输出 HTML，优先呈现首屏骨架
- 选型对照：SSE（多用于 LLM 单向逐字）vs WebSocket（双向实时，后续 D53 展开）
- 产出：一个用 SSE/ReadableStream 实现逐块输出的流式 UI 示例

**检查点**：
- [ ] 能解释 SSE 相比轮询和 WebSocket 在 LLM 流式场景的取舍
- [ ] 能用 ReadableStream + reader 在前端逐块消费并渲染服务端推流
- [ ] 理解 Suspense 边界如何配合流式 SSR 优先呈现骨架

## Week 7 (D43-D49)：AI 对话界面核心

### D43 周一：Vercel AI SDK（useChat/useCompletion/streamText）（2h）

**学习内容**：

- 产出：支持逐字输出 + Markdown + 代码高亮的流式渲染组件

**检查点**：

### D45 周三：对话状态管理（消息列表/多会话/Zustand）（2h）

**学习内容**：

- 会话与 useChat 协同：切换会话时加载对应消息，新建会话重置对话上下文
- 状态规范化：消息按 id 索引、会话列表与消息分离存储，便于增量更新
- 与服务端的同步策略：本地状态先行更新（乐观更新）+ 后端持久化（为 D49 持久化铺垫）
- 产出：基于 Zustand 的多会话对话状态管理

**检查点**：
